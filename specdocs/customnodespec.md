# Spec: User-defined Custom Nodes (sandbox)

A node type whose `compute` is JavaScript the user wrote, not TypeScript
that ships with the build. The user opens an editor in-app, writes a
function, hits run — the new behavior shows up immediately on the
canvas. The node and its source live in the project, save/load with it,
and round-trip through the same JSON envelope as everything else.

This is the "real customization" path that fits the existing
architecture — no in-browser TS transpiler, no hot-swap of built-in
nodes, no live-reload from disk.

---

## 0. Current state to build on

- Node registry: `src/nodes/index.ts` calls `registerNode(def)` per
  built-in node. `getNodeDef(type)` looks defs up by type string.
- Graph schema: `NodeDataPayload` (in `src/state/graph.ts`) carries
  `defType`, `params`, `exposedParams`, etc. Adding new fields is
  cheap — they round-trip through `serializeGraph` /
  `deserializeGraph` in `src/lib/project.ts` automatically because
  the unknown-key branch is `out[key] = val`.
- Evaluator: walks the DAG, calls `def.compute({ inputs, params, ctx,
  nodeId })`. A def with a misbehaving `compute` already gets caught
  per-node — errors don't kill the run.
- ParamPanel already renders bespoke editors for `paint`, `curves`,
  `color_ramp`, `timeline_curve`, `merge_layers`. A "code" param type
  fits the same pattern.
- No globals.css / Monaco / sandbox infra exists yet. Either is a
  net-new dependency.

---

## 1. Scope

**In:**
- A new built-in node type `custom-script` with a `code` param of
  type `script`.
- An in-app editor (Monaco or CodeMirror) that opens on a tab in the
  canvas dock area (parallel to the timeline curve editor).
- A constrained JS runtime — no DOM, no fetch, no `import`, no
  `WebGLRenderingContext` — that runs the user's compute every frame.
- A small declarative API for inputs, outputs, and params: the user
  declares them at the top of the script and the editor builds a
  matching node shape (sockets + ParamPanel rows).
- Project save/load round-trip. The script is part of the project JSON.
- Error surface: compile errors + runtime errors render on the node
  with a hover tooltip carrying the message + line number.

**Out (v1):**
- WebGL / fragment-shader access from inside the sandbox. Custom
  nodes are CPU-only on `scalar`/`vec*`/`points`/`spline`. No
  texture writes.
- Cross-node imports / shared user libraries.
- Per-node persistent state across runs (no `init` / `dispose` —
  pure functions only).
- Sharing custom nodes between projects (see §10 future work).
- Async compute. `compute` must be synchronous.

---

## 2. The author's experience

1. User adds a **Custom Node** from the Shift+A palette.
2. The node lands on the canvas with one default scalar input and
   one scalar output, named "Custom 1".
3. User selects the node. ParamPanel shows the standard rows plus a
   single `code` row with a "Edit script" button. (Following the
   timeline-curve pattern: thumbnail in the panel, full editor docked
   to the canvas via a tab.)
4. Clicking the tab opens a Monaco editor over the bottom of the
   canvas area (mirrors the timeline editor placement). The script
   is on the left; an autogenerated "Inputs / Outputs / Params"
   declaration panel and a live error list are on the right.
5. Edits compile-on-stop (debounced ~300ms). The graph re-evaluates
   automatically — no separate "run" button.
6. Closing the tab returns to the canvas.

The intent: feels like editing a curve in the timeline editor.
Same dock, same close affordance.

## 3. Script shape

A custom node is a single JS file whose top-level `export default` is
a plain object literal — call it the **NodeSpec**. Same shape as the
built-in `NodeDefinition` but with the unsafe knobs removed.

```js
export default {
  // Free-form display name. Renders on the node header.
  name: "My Custom Node",

  // Sockets the user can wire into. Type strings are the same union
  // as the built-in `SocketType`. Required defaults to false.
  inputs: [
    { name: "in", type: "scalar", required: true },
    { name: "factor", type: "scalar" },
  ],

  // Sockets that come out of the node. The first entry is the
  // primary; remaining entries become aux outputs.
  outputs: [
    { name: "out", type: "scalar" },
    { name: "doubled", type: "scalar" },
  ],

  // Same param-def shape as built-in nodes (subset of types — see §6).
  params: [
    { name: "amount", type: "scalar", min: 0, max: 1, default: 0.5 },
  ],

  // Pure function. Receives resolved inputs and the param map.
  // Returns an object with the same shape as `outputs`. Throwing
  // raises a per-node error without breaking other nodes' eval.
  compute({ inputs, params }) {
    const a = inputs.in?.value ?? 0;
    const f = inputs.factor?.value ?? 1;
    const m = params.amount;
    return {
      out: { kind: "scalar", value: a * f * m },
      doubled: { kind: "scalar", value: a * 2 },
    };
  },
};
```

**Design notes:**
- `compute` is a plain function. The runtime extracts `inputs`,
  `params` from the args; the user doesn't need to know about `ctx`,
  `nodeId`, `auxIn` (the bits we strip out).
- Inputs / outputs / params are declared inline so the editor can
  parse them and build the node shape at the moment the script is
  saved. No separate "register sockets" step.
- The runtime sees the export, not the source — see §4.

## 4. Sandbox

The script is compiled and executed inside an isolated context. We
do **not** use `eval` or bare `new Function(...)` on the main thread.

**Recommended runtime: [QuickJS-WASM](https://github.com/justjake/quickjs-emscripten).**

Reasoning:
- Real V8/JS sandbox running inside WebAssembly. The user can't reach
  `window`, `document`, `fetch`, or anything DOM/network from the
  script.
- Synchronous interpreter — fits the per-frame `compute` contract.
- Cheap enough to run per-frame for a small number of custom nodes
  (the typical scene won't have dozens).
- Crashes / infinite loops can be killed via the QuickJS interrupt
  handler. We bound each `compute` call to ~16ms of execution; if the
  script runs over, we abort it and surface a "compute timed out"
  error on the node.

**Alternative: `new Function(...)` + Realms shim.** Lighter weight,
no WASM bundle (~400KB), but you can't kill a runaway loop and the
sandbox is much weaker (the script can crawl the global scope unless
you carefully snapshot and restore). Acceptable fallback for v1 if
the QuickJS bundle is judged too heavy; revisit at v2.

**What's exposed to the script** — only these globals are visible:

| Global  | Notes                                                       |
|---------|-------------------------------------------------------------|
| `Math`  | Full standard library.                                      |
| `JSON`  | parse / stringify only.                                     |
| `console.log` / `warn` / `error` | Routed to the editor's console pane. |
| `Number`, `Array`, `String`, `Object`, `Boolean` | Standard prototypes. |

Nothing else. No `setTimeout`, no `Promise`, no `import`, no `fetch`.

## 5. Type surface for inputs / outputs

`SocketType` reuses the existing union: `image`, `mask`, `uv`,
`scalar`, `vec2`, `vec3`, `vec4`, `spline`, `points`, `audio`,
`image_group`.

**v1 restriction**: custom nodes can produce or consume only the
CPU-native types — `scalar`, `vec2`, `vec3`, `vec4`, `spline`,
`points`. Image/mask/UV/audio are blocked at the editor level (the
type dropdown in the inputs/outputs declaration won't list them) and
also at compile time (the runtime rejects a `NodeSpec` that declares
them with a clear error).

Reasoning: image/mask/UV are GPU textures wrapped in tiny JS objects
referencing WebGL handles. Exposing them to the sandbox means either
(a) leaking GL state through the QuickJS bridge, which defeats the
sandbox, or (b) reading them back into typed arrays per frame, which
is an order of magnitude too slow. Audio carries live HTMLAudioElements
— same problem. Defer all of these to a "GPU custom node" follow-up.

For the allowed types, the runtime serializes the input on the way in
to plain JS values:

```ts
// scalar  → { kind: "scalar", value: number }
// vec2    → { kind: "vec2", value: [number, number] }
// spline  → { kind: "spline", subpaths: [...] }      // structurally cloned
// points  → { kind: "points", points: [...] }         // structurally cloned
```

The user reads them as if they were the engine's own values. On the
way back, the same shapes are parsed and validated — anything that
doesn't conform is logged as a per-frame "output type mismatch" error
and the prior frame's output is reused.

---

## 6. ParamType surface

Custom nodes can declare params of these types only:

- `scalar`, `vec2`, `vec3`, `vec4`, `color`, `boolean`
- `enum` (with `options: string[]`)
- `string`

That covers everything ParamPanel can render generically. The
authoring-heavy types (`paint`, `color_ramp`, `curves`,
`timeline_curve`, `merge_layers`, `spline_anchors`, file types) all
have bespoke React UIs and don't make sense as v1 declarations from
script — defer.

The custom node's `params` list is fed straight into the existing
`exposedParams` plumbing — meaning users can **expose** any custom
param as a socket, just like any built-in node. Free.

---

## 7. Where the script lives

A new `ParamType`: `"script"`. Value shape:

```ts
type ScriptParamValue = {
  source: string;     // raw JS the user typed
  spec?: NodeSpec;    // last successfully compiled spec
  // Optional cached compile output. Always re-derivable from `source`,
  // so we don't have to persist it — keeps save files lean. Stored
  // only in the live ParamValue, not the JSON envelope.
};
```

The custom node is a single built-in `NodeDefinition`:

```ts
{
  type: "custom-script",
  name: "Custom Node",
  category: "utility",
  params: [
    { name: "code", type: "script", default: { source: DEFAULT_SCRIPT } },
  ],
  // resolveInputs / resolvePrimaryOutput / resolveAuxOutputs all
  // pull from params.code.spec.inputs/outputs, falling back to a
  // single scalar in / scalar out when the script hasn't compiled
  // yet (or has errored).
  resolveInputs(params) { … },
  resolvePrimaryOutput(params) { … },
  resolveAuxOutputs(params) { … },
  compute({ inputs, params, ctx, nodeId }) {
    // Look up the compiled QuickJS function for this nodeId from
    // ctx.state, or compile it now if `params.code.source` has
    // changed since last call. Then call it with serialized inputs.
  },
  dispose(ctx, nodeId) {
    // Free the QuickJS context held in ctx.state.
  },
}
```

The resolveInputs / resolvePrimaryOutput / resolveAuxOutputs functions
read `params.code.spec` and translate the user-declared sockets into
the engine's `InputSocketDef[]` / `SocketType` shape. When the script
hasn't compiled yet, fall back to a default of one scalar in and one
scalar out — gives the user a working node from the moment of drop.

A handful of evaluator / param-panel touchpoints already key off
`def.type` for special cases (paint, etc.) — they need to learn about
`custom-script` too. Keep that list short; the existing polymorphism
via `resolveInputs` etc. covers most of it.

## 8. The editor UI

**Component**: `CustomScriptEditor.tsx`, sibling of `TimelineCurveEditor`.

**Placement**: same dock pattern. A tab labelled `</> script` pokes
up from the bottom-center of the canvas area when a `custom-script`
node is selected. Clicking opens it; clicking the close button
collapses it back to the tab. State per-node like the timeline editor.

**Layout** (300px tall default, user-resizable):

- Left pane (~70%): Monaco editor in `javascript` mode. Syntax
  highlighting, autocomplete on the standard library + the
  `inputs`/`params` argument shape (we feed Monaco a TS `.d.ts`
  describing the API).
- Right pane (~30%): two stacked sections.
  - **Inputs / Outputs / Params**: read-only — derived from the last
    successful compile. Helps the user remember what they declared.
  - **Console**: `console.log` from the script renders here, oldest
    at top. Plus the latest compile or runtime error with line/col.

**Compile cadence**: debounce 300ms after the last keystroke. On
success, atomically swap the node's spec; on failure, leave the
previous spec in place so the canvas keeps rendering whatever it was
before.

**ParamPanel integration**: a `script` ParamType row renders a tiny
preview chip ("Custom Node · 12 lines") plus an "Edit" button that
opens the dock. Same pattern as the timeline curve thumbnail.

## 9. Serialization

`script` ParamValues serialize as the `source` string only. The
`spec` field is rebuilt on load by re-running the user's script
through the sandbox.

Custom nodes round-trip through the same `serializeGraph` /
`deserializeGraph` paths as everything else — we only need to add a
`script` branch to `serializeParams` / `deserializeParams` in
`src/lib/project.ts`, and even that's optional (the default `out[key]
= val` branch handles it correctly because `source` is a plain
string).

The compiled QuickJS context is process-local and intentionally not
persisted. Loading a saved project recompiles every custom node's
script from `source`. Compile errors at load time render the node in
its error state; the project still loads.

## 10. Error model

Three error tiers, each surfaced on the node header (existing `data.error`
field):

1. **Compile error** (script doesn't parse): node shows "ERR" red
   pill, hover tooltip has "Parse error: <msg> at line N". The node's
   sockets fall back to the default 1-in / 1-out shape so wires from
   prior versions of the node don't get orphaned.
2. **Spec error** (script parses but the exported `NodeSpec` is
   invalid — e.g. a disallowed socket type, malformed inputs array):
   node shows "ERR", tooltip carries the specific validation
   message. Sockets fall back to default.
3. **Runtime error** (script throws or times out during `compute`):
   node renders the error pill, tooltip shows the throw message. The
   node returns the previous frame's output if available, otherwise
   the type's zero-value (0 for scalar, [0,0] for vec2, [] for
   points, etc.) so downstream nodes don't crash on `undefined`.

A timeout (>16ms in `compute`) is treated as a runtime error and the
script's QuickJS context is reset, so a bad infinite loop doesn't
poison subsequent frames.

## 11. Performance

- One QuickJS context per node instance. Reused across frames; reset
  only on script edit, on timeout, or on node disposal.
- `compute` overhead is dominated by (a) serializing inputs into
  QuickJS, and (b) deserializing outputs back. For scalar / vec
  types this is microseconds. For points / spline it depends on
  cardinality — a 10k-point input is roughly the cost of structured-
  cloning a 10k-element array, which is fine for animation rates
  (sub-millisecond).
- The QuickJS bundle itself is ~400KB gzipped. Lazy-load it only
  when the first custom node is added or loaded. Until then the
  payload cost is zero.
- We do **not** memoize compute results inside the sandbox; the
  evaluator's outer fingerprint cache already does that. Don't
  re-implement caching here.

## 12. Security

Even with QuickJS isolating the script, treat user scripts as
**untrusted code that runs in the user's own browser only**.

Specifically:
- Custom nodes never execute on the server. Saved projects store the
  source string; we never `eval` it server-side, never include it in
  a Next.js loader, never feed it to an SSR pass.
- A project loaded from another user (public projects, see
  `lib/supabase/projects.ts`) executes its custom nodes immediately
  on the loader's machine. The QuickJS sandbox is what makes that
  safe — without it, untrusted JS reading another user's localStorage
  or making fetch calls would be a real risk.
- We document this on the public-projects flow: "Public projects can
  contain custom code that runs in your browser. We sandbox it, but
  scrutinize anything the script logs to the console before trusting
  it." Add a one-time confirm modal the first time a user loads a
  public project containing custom nodes.

## 13. Implementation milestones

A reasonable order. Each milestone leaves a working app.

1. **`script` ParamType + `custom-script` node skeleton.** Default
   socket shape (1 scalar in / 1 scalar out). `compute` is a stub
   that just passes input → output. Saves and loads correctly. No
   editor yet — the source lives in the saved JSON. Validates the
   schema plumbing.
2. **QuickJS integration, no editor.** `compute` lazy-compiles
   `params.code.source` and runs it through QuickJS. Hard-coded
   default script that doubles its input. Confirms the runtime works
   per-frame at a usable speed.
3. **Resolve sockets/params from the spec.** `resolveInputs`,
   `resolvePrimaryOutput`, `resolveAuxOutputs` read from the compiled
   spec. Editing the source by hand-editing the JSON file changes
   the node's shape on reload.
4. **In-app Monaco editor + dock UI.** Edits debounce-compile, the
   spec live-updates, errors surface on the node. Mirrors the
   timeline curve editor's placement.
5. **TS `.d.ts` for autocomplete + console pane.** Polish layer.
6. **Public-project confirm modal.** Ships with milestone 1's
   schema or earlier — but only matters once someone publishes a
   project containing a custom node.

## 14. Open questions

- **Editor library**: Monaco is the obvious pick (full IntelliSense,
  TS support if we ever want it later) but adds ~2MB. CodeMirror 6
  is ~200KB and fine for plain JS. Monaco wins on user experience;
  CodeMirror wins on bundle size. Default to Monaco loaded
  dynamically only when the editor opens — same pattern we'd use
  for QuickJS — so the cost is paid by the user who needs it.
- **Multiple custom nodes sharing code**: deferred. v1 is one script
  per node. If users start copying scripts between nodes, add a
  "library" concept later.
- **Rate-limiting compile-on-type**: 300ms debounce should be plenty,
  but a script that does heavy work in its top level (rare for a
  pure spec export but possible) would block the UI on every
  recompile. If that becomes a problem, run the compile in a Web
  Worker. Don't pre-emptively engineer it.
- **Per-frame `console.log` flooding**: a script that logs every
  frame will fill the console pane in seconds. Throttle to last N
  unique messages; suggest `console.warn` for one-off diagnostics.
- **GPU custom nodes** are a separate spec doc. The data flow
  (texture in → texture out) and the API surface (a fragment-shader
  string + uniform declarations) are different enough that bolting
  them onto this design would muddy it. Park until v2.

---

## 15. Things explicitly NOT in this spec

- Editing built-in nodes' source from inside the app. (Out of scope —
  see the conversation that spawned this doc.)
- Hot-reloading the dev server based on user edits.
- A marketplace / sharing system for custom nodes.
- Custom param renderers (the user can't define a new ParamType from
  inside a custom node).
- Async / streaming computes.
- Any form of network access from the script. Ever.
